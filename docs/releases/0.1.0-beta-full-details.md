# RL Agent Plugin 0.1.0-beta Full Technical Release Notes

This document is the technical release note for version 0.1.0-beta of RL Agent Plugin for Godot.

It is intended for engineers who want implementation-level details, architecture context, and operational behavior beyond the short public release note.

## Release Scope

Version 0.1.0-beta delivers an end-to-end reinforcement learning workflow inside Godot 4.6+ with C#:

- Author RL environments directly in Godot scenes
- Train PPO and SAC policies in-editor
- Monitor live metrics in RLDash
- Scale rollout collection with distributed workers
- Export checkpoints to rlmodel for runtime inference

## System Overview

At a high level, the plugin consists of:

- Runtime agent API for 2D and 3D tasks
- Training bootstrap and trainer stack
- Config resources for algorithms, run behavior, curriculum, self-play, and distributed execution
- Editor tooling for launch, validation, and metrics visualization
- Checkpoint and model export/import pipeline for deployment

Core scene pattern:

- TrainingScene
- RLAcademy node as training coordinator
- One or more RLAgent2D or RLAgent3D nodes under the environment

## Core Runtime and Agent Lifecycle

RL agents implement a lifecycle centered on:

- DefineActions
- CollectObservations
- OnActionsReceived
- OnStep
- OnEpisodeBegin

Optional hooks:

- OnTrainingProgress for curriculum integration
- OnHumanInput for manual control workflows

Operational flow per physics step in training mode:

1. Observations are collected from each active agent.
2. Policy actions are sampled for each policy group.
3. Actions are applied to simulation objects.
4. Reward accumulation and episode termination checks run.
5. Transitions are recorded to trainer buffers.
6. Updates run once trainer-specific buffer criteria are met.

## Algorithms

### PPO

- Supports discrete and continuous action spaces.
- Uses clipped policy updates with value learning and entropy regularization.
- Typical controls include rollout length, epochs per update, learning rate, clip epsilon, entropy coefficient, and GAE parameters.

Best fit:

- Discrete-action tasks
- Fast baseline iteration
- Stable first-pass training setups

### SAC

- Supports continuous action spaces.
- Uses replay buffer, double critics, target networks, and automatic entropy temperature tuning.
- Typical controls include replay capacity, batch size, warmup steps, tau, learning rate, update frequency, and updates-per-step.

Best fit:

- Continuous control
- Sample-efficiency-sensitive tasks
- Higher-complexity locomotion/control environments

## Policy Grouping and Multi-Agent Training

Policy groups allow weight sharing and multi-policy layouts:

- Agents with the same group identifier share one policy/trainer.
- Different group identifiers train separate policies.

Self-play support includes:

- Policy pairing definitions
- Historical checkpoint pools
- Optional PFSP sampling
- Elo tracking for competitive progress monitoring

This enables robust adversarial training loops for multi-agent scenarios.

## Observation and Sensor Pipeline

Supported observation patterns:

- Scalar and vector observations
- Normalized observation helpers
- Sensor-based observation components
- Image streams for camera-driven policies

Built-in workflows include raycast and transform/velocity style sensing patterns, with support for extending observation logic in custom agent code.

For image observations, encoder paths are built automatically as part of policy network construction.

## Training Execution and Run Behavior

Run-level execution is controlled through RLRunConfig and related resources.

Key behaviors include:

- BatchSize environment replication in the training bootstrap
- Simulation speed control for faster-than-real-time training
- Action repeat control for temporal abstraction
- Checkpoint interval configuration
- Optional asynchronous gradient updates
- Parallel policy group processing options

This allows tuning between debug visibility, throughput, and training stability.

## Distributed Rollout Architecture

Distributed mode adds headless rollout workers connected to a master process:

- Master owns trainer state and weight updates.
- Workers run environment simulation and collect transitions.
- Workers send rollouts to master.
- Master updates policy and broadcasts fresh weights.

Practical outcomes:

- Increased data collection throughput
- Better utilization for complex environments
- Faster wall-clock iteration for large training runs

## GPU CNN Training Path for PPO Image Workloads

When all activation conditions are met:

- At least one image observation stream exists
- Active algorithm is PPO
- Vulkan compute is available

The plugin can enable a GPU-backed CNN training path for image encoder updates.

Current execution split:

- Rollout collection and action sampling remain CPU-side
- Image encoder training updates can run on GPU
- Shared trunk and policy/value heads remain CPU-side

This targets the highest-cost part of camera-heavy PPO training while preserving compatibility with existing runtime flow.

## Metrics, Logging, and Dashboard

Training writes structured metrics in run folders under RL-Agent-Training.

Common tracked fields include:

- Episode reward
- Episode length
- Policy and value loss
- Entropy
- Total steps and episode count
- Policy group identity
- Curriculum progress
- Self-play indicators such as Elo where applicable

RLDash reads metrics files and renders live charts for real-time training diagnosis.

## Model and Checkpoint Pipeline

The plugin supports:

- Periodic checkpoint persistence during training
- Export to rlmodel for deployment and inference workflows
- Loading models back through policy group inference configuration

This enables a train-then-deploy loop inside the same Godot project setup.

## Editor Tooling Surface

Primary editor-facing controls:

- Top toolbar actions for Start Training, Stop Training, and Run Inference
- RL Setup dock for setup and launch workflow
- RLDash for run monitoring and model export operations

This keeps operational tasks inside editor UX rather than requiring external orchestration scripts for common workflows.

## Demonstration Environments

The companion workspace provides representative environment coverage:

- 2D single-agent reach target
- 2D multi-agent self-play tag
- 3D curriculum wall climb
- 3D move-to-target continuous control
- 3D crawler locomotion
- Ball tracker style image-observation scenario

These scenarios demonstrate progression from baseline setup to advanced RL patterns.

## Known Beta Boundaries

Current scope is intentionally beta-grade and still evolving.

Important boundaries:

- Feature and UX polish continue beyond this release.
- Some advanced reliability and reproducibility capabilities are planned improvements, not final guarantees.
- GPU acceleration scope currently focuses on PPO image encoder training path rather than full-network GPU execution.

## Recommended Adoption Pattern

For teams integrating 0.1.0-beta:

1. Start from a minimal demo-equivalent environment.
2. Validate observation normalization and reward scale early.
3. Establish baseline PPO training first.
4. Move to SAC for difficult continuous-control workloads.
5. Introduce distributed workers when rollout collection becomes bottleneck.
6. Export and validate inference behavior in gameplay scenes before content lock.

## Summary

0.1.0-beta establishes the foundational architecture for Godot-native reinforcement learning workflows:

- In-editor training and monitoring
- Practical algorithm coverage with PPO and SAC
- Advanced methods including curriculum and self-play
- Distributed rollouts for scalability
- Deployable model export for inference

This release is intended as a production-oriented beta foundation and a platform for iterative hardening in upcoming versions.
